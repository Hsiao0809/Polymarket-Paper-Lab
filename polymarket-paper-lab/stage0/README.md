# 階段 0 + 1：資料採集與 Gap 量測

> **這個工具不會下單，也不會碰你的錢。** 它只做兩件事：記錄資料、告訴你 edge 存不存在。
> 本專案不構成投資建議。

---

## 這是在做什麼

Polymarket 的 `SPY Daily Up or Down` 市場，結算條件是「今日收盤 > 昨收」。
昨收是**已知固定值**，而市場只在正常交易時段（13:30–20:00 UTC）交易。

這代表該合約 = **履約價為昨收、到期為今日收盤的數位買權**。

而 SPY 每個交易日都有 **0DTE 期權，到期時點完全一致**。所以：

```
P(收盤 > 昨收) = -∂C/∂K ≈ [C(K-h) - C(K+h)] / 2h
```

**直接從期權鏈讀出機率，不需要估計任何波動率參數。**

這比先估波動率再套 Black-Scholes 穩健：不必選用哪段波動率、不必處理波動率
微笑、少一個會出錯的自由參數。也因為市場只在盤中交易，**根本不存在「日盤/
夜盤波動率如何混合」的問題**——隔夜跳空在開盤時就已實現完畢。

然後拿這個機率去比對 Polymarket 中價，量測 gap。

## 使用

### 1. 先驗證邏輯（不需網路）

```bash
python test_pricing.py
```

應顯示 `15/15 通過`。**這一步不過就不要往下走**——資料採再多也沒意義。

測試內容包含：數位機率導出 vs BS 解析解（誤差 <0.001）、單調性、
資料品質防護、手續費公式、實測盤口試算。

### 2. 測試接線

```bash
python collector.py --once
```

預期輸出類似：

```
[14:32:01Z]  PM 0.76/0.77 mid=0.765  模型=0.782 (call_spread)  gap=+0.017 費用=0.009  ← 扣費後仍有優勢
```

### 3. 持續採集

在美股交易時段（13:30–20:00 UTC）執行：

```bash
python collector.py
```

每 60 秒一筆，`Ctrl+C` 停止。建議至少累積 **5 個完整交易日**再看結論。

### 4. 分析

```bash
python analyze.py
python analyze.py --plot gap.png
```

## 怎麼讀分析結果

報告最後會給一個判斷。**請注意哪個結果才是「好消息」：**

| 有優勢時點佔比 | 判讀 |
|---|---|
| **< 5%** | **吃單腿死了。這是好結果**——你在花錢之前就知道了。只剩做市可走 |
| 5–20% | 邊際可行但脆弱。未計延遲與滑價，實際更差 |
| **> 20%** | **先懷疑資料，不要興奮。** yfinance 有 ~15 分鐘延遲，會製造假 gap |

最後那一列要特別小心。**延遲資料製造出來的 gap 是最常見的假陽性來源**——
你看到的「機會」其實只是你在看過去的價格。要採信必須換即時報價源重測。

## 已內建的兩個關鍵發現

### 1. Gamma API 的報價會延遲，不能用

2026-07-21 實測，同一時刻：

| 來源 | bestBid / bestAsk | 價差 |
|---|---|---|
| Gamma API | 0.76 / 0.79 | 3¢ |
| **CLOB order book** | **0.76 / 0.77** | **1¢** |

用 Gamma 報價做研究會**系統性高估價差、高估獲利空間**。
本專案一律讀 CLOB book，Gamma 欄位僅存作對照（`gamma_bid` / `gamma_ask`）。

### 2. 手續費門檻在極端價位低很多

費用公式為 `fee = 0.04 × min(p, 1-p)`，所以：

| 中價 | 所需 gap 門檻 |
|---|---|
| 0.50 | **2.00¢** |
| 0.65 | 1.40¢ |
| 0.765 | 0.94¢ |
| 0.90 | 0.40¢ |
| 0.95 | **0.20¢** |

**機會不是均勻分布的。** 越接近 0 或 1，越小的定價誤差就足以獲利。
這也意味著在價平附近吃單幾乎不可能划算——2¢ 的門檻比典型價差還寬。

## 雲端執行（不用一直開電腦）

用 **GitHub Actions** 排程，免費、原生跑 Python、不需要 VPS。

### 設定步驟

1. 在 GitHub 建一個 repo（**建議設為 Public**，理由見下）
2. 把整個 `polymarket-paper-lab` 資料夾推上去，含 `.github/workflows/collect.yml`
3. 進 repo 的 **Settings → Actions → General → Workflow permissions**，
   選 **Read and write permissions**（否則 bot 無法把資料 commit 回來）
4. 進 **Actions** 分頁 → 選 `Collect Polymarket data` → **Run workflow** 手動測一次
5. 確認 repo 出現 `data/snapshots.jsonl` 且內容正常，之後就會自動每 10 分鐘跑

### 拉回來分析

```bash
git pull
cd stage0
python import_jsonl.py ../data/snapshots.jsonl
python analyze.py --plot gap.png
```

### 額度與限制

| 項目 | 說明 |
|---|---|
| 私有 repo | 免費 2,000 分鐘/月。本設定約用 **560 分鐘/月**，夠用 |
| 公開 repo | **完全不計額度**，所以建議用 Public |
| 排程精準度 | GitHub 排程**不保證準時**，尖峰可能延遲 5–20 分鐘 |
| 自動停用 | repo **連續 60 天無活動**，排程會被自動停用。每隔一陣子推個 commit 即可 |
| 單次上限 | 6 小時（本工作約 40 秒，無虞）|

Public repo 的代價是**採集到的資料公開可見**。這裡只有公開市場報價，
沒有任何帳號或金鑰資訊，所以無妨——但**永遠不要把 API key 放進 repo**。

### ⚠ 這個設定能回答什麼、不能回答什麼

這是整個雲端方案最重要的一段。

**採樣間隔 10 分鐘 + yfinance 約 15 分鐘延遲 = 資料整體落後約 15–25 分鐘。**

| 問題 | 能否回答 |
|---|---|
| 是否存在**持續性、大幅**的定價落差？ | ✅ 可以 |
| gap 的分布、時段分布、隨到期時間的變化？ | ✅ 可以 |
| 做市該掛在哪個價格區間比較安全？ | ✅ 大致可以 |
| 能不能抓到 1¢ 的瞬間價差？ | ❌ **完全不能** |
| 延遲敏感的策略是否可行？ | ❌ **完全不能** |

換句話說：**這套設定是用來篩掉壞主意的，不是用來驗證好主意的。**

如果分析顯示「有優勢時點 > 20%」，那幾乎可以確定是延遲造成的假象——
你看到的「機會」其實是拿舊的期權價去比新的 Polymarket 價。
真要採信，必須換即時報價源，而那要花錢（貼文作者就是走到這一步才付費）。

**先用免費方案確認方向，有把握了再花錢。順序不要反。**

## 已知限制

- **yfinance 有約 15 分鐘延遲**。足夠做研究，不足以實盤報價
- **結算源不一致**：Polymarket 用 Pyth，本工具用 Yahoo。通常極接近但不保證相同
- **尚未模擬逆選擇**。那是做市的主要風險，屬階段 2
- 非交易日 / 無 0DTE 時會標記 `quality` 並在分析中排除
- 獎勵池份額無法從公開盤口反推，需實際掛單後觀察

## 下一步（階段 2）

只有在階段 1 顯示值得繼續時才做：

1. 以模型公允值為中心模擬雙邊掛單
2. 保守成交假設（市價穿過才成交）
3. **必須模擬逆選擇**：算成交後到結算的實際盈虧，而非假設賺到價差
4. 拆解「價差收益 / 獎勵收益 / 逆選擇損失」三項

逆選擇是做市唯一的真風險：你的掛單總是在你判斷錯誤的那一刻成交。
不模擬它的回測一定是騙人的。
